# 可靠性加固设计

## 背景

当前项目已经形成台账导入、稽核、问题包导出、整改回传、专题分析和归档的完整本地业务闭环。现有 145 个自动化测试全部通过，当前主要风险已从功能缺失转为数据升级、备份恢复、重复稽核、并发重操作和持续集成不足。

本次工作保留 Python 标准库 HTTP 服务、SQLite、原生 ES Modules 和 PyInstaller 单文件交付方式，不引入新的运行时框架，不改变现有成功 API 响应和业务操作流程。

## 目标

- 建立可追踪、可回滚、可测试的顺序数据库迁移体系。
- 使用 SQLite 原生能力生成并校验一致性备份，安全恢复损坏前的数据。
- 明确定义重复稽核后当前问题与历史结果的关系。
- 防止重操作并发执行和超大上传占满进程内存。
- 建立覆盖后端、前端模块和启动脚本的 PR 持续集成门禁。

## 非目标

- 不迁移到 FastAPI、SQLAlchemy、React 或其他新框架。
- 不引入多用户账号、联网协作、任务队列或外部数据库。
- 不拆分全部大型模块；只允许为本次可靠性能力建立清晰边界所需的局部调整。
- 不改变现有 Excel 模板、导出格式、规则口径或专题分析金额口径。
- 不增加新的业务页面。

## 总体架构

采用渐进式原地加固：

- `db.py` 保留连接管理和数据库初始化入口。
- 新增 `migrations.py`，集中声明并顺序执行版本化迁移。
- `backup.py` 使用 SQLite Backup API 和完整性校验，不再直接复制活动数据库。
- `audit_engine.py` 在每次稽核后同步当前问题，并保留每次 `audit_runs` 与 `audit_results` 历史。
- 新增进程内操作协调器，为破坏性或高成本操作提供互斥保护。
- `server.py` 在读取上传正文前检查长度，并把操作冲突映射为 HTTP 409、超限映射为 HTTP 413。
- 新增质量检查工作流，在 Pull Request 和主分支推送时运行统一检查脚本。

## 数据库迁移

### 版本模型

`schema_migrations` 继续作为版本事实来源，每个成功应用的迁移写入一行。迁移以严格递增整数标识，应用按照版本排序，只执行尚未记录的迁移。

现有数据库结构视为版本 1。全新数据库先创建版本 1 基础结构，再顺序升级到当前版本。若旧数据库没有 `schema_migrations`，初始化器先识别现有核心表；存在核心表时登记为版本 1，不存在时创建版本 1。

本次版本 2 迁移增加支持重复稽核所需的字段和状态历史表：

- `issues.resolved_at`：本次稽核不再命中时的解除时间。
- `issues.last_seen_audit_run_id`：最近一次命中的稽核运行。
- `issue_events`：记录问题状态变化、来源操作、变化时间和可选说明。

`issues.status` 新增 `resolved_by_reaudit`。旧人工状态、整改值和整改说明不因重复稽核被清除。

### 执行规则

- 每个迁移在独立事务中执行。
- 已存在的数据库如有待执行迁移，先通过 SQLite Backup API 在 `backups/` 生成并校验一次升级前备份；备份失败则不执行迁移。
- 迁移成功后才写入对应版本。
- 迁移失败时回滚该迁移并阻止应用启动。
- 数据库记录的最高版本高于应用支持版本时拒绝启动，避免旧程序打开新数据库。
- 重复初始化必须幂等。
- 迁移入口保持在 `initialize_database`，调用方无需改变。

## 备份与恢复

### 备份

备份通过源连接的 SQLite `backup` 方法写入目标数据库。写入完成后，使用独立只读连接执行 `PRAGMA integrity_check`；结果不是 `ok` 时删除目标文件并返回错误。

备份文件仍保存在工作区 `backups/`，命名规则保持不变。备份创建过程不要求关闭本地服务。

### 恢复

恢复按以下顺序执行：

1. 验证来源位于 `backups/` 且为普通文件。
2. 对来源执行完整性检查。
3. 检查来源数据库版本不高于当前应用支持版本。
4. 使用 SQLite Backup API 创建当前数据库安全备份。
5. 在操作锁保护下将来源数据库恢复到活动数据库。
6. 重新打开活动数据库并执行完整性检查和迁移初始化。

来源校验失败时不得改动活动数据库。恢复失败时使用刚生成的安全备份还原原数据库，并向调用方返回明确错误。

## 重复稽核一致性

每次稽核继续创建新的 `audit_runs` 和 `audit_results`，作为不可覆盖的运行历史。问题使用由批次、台账行和规则组成的稳定问题编号同步：

- 首次命中：新增问题，状态为 `pending_export`。
- 持续命中：更新 `audit_result_id`、严重程度、问题说明、建议和 `last_seen_audit_run_id`，保留当前人工状态、整改值和整改说明。
- 曾由重复稽核解除后再次命中：重新置为 `pending_export`，清空 `resolved_at`，写入重新打开事件；历史整改说明仍保留。
- 本次未命中且此前仍属于活动问题：置为 `resolved_by_reaudit`，写入 `resolved_at` 和状态事件。
- 已归档批次：继续禁止重新稽核。

“活动问题”指除 `resolved_by_reaudit` 外的现有问题状态。专题分析只读取本次有效命中问题，避免已解除问题继续生成金额线索。

## 操作互斥与上传限制

新增进程内操作协调器，以工作区为粒度管理一个非阻塞互斥锁。以下操作必须在锁内执行：

- 正式导入与覆盖/追加导入。
- 执行稽核。
- 备份恢复。
- 系统复位。
- 数据库维护压缩。
- 批次归档。

锁已被占用时不排队，API 返回 HTTP 409 和“系统正在执行其他数据操作，请稍后重试”。无论操作成功或抛出异常，锁都必须释放。只读查询、预检和导出不占用全局锁。

上传上限定义为服务端常量，第一版为 100 MiB。请求缺少或包含非法 `Content-Length` 时返回 HTTP 400；长度超过限制时在读取正文前返回 HTTP 413。文件扩展名和空文件校验继续沿用现有逻辑。

## API 与错误处理

现有成功响应保持不变。新增错误约定：

- 数据操作冲突：HTTP 409，JSON `error` 为明确中文说明。
- 上传超限：HTTP 413，说明允许的最大文件大小。
- 非法请求长度：HTTP 400。
- 损坏备份、迁移失败和版本不兼容：HTTP 400；错误说明包含可执行的处理建议，但不暴露堆栈。

数据库初始化失败发生在服务监听端口之前，进程打印错误并退出，避免在不兼容数据库上继续提供部分功能。

## 持续集成与本地检查

扩展 `scripts/check.sh`：

- 运行全部 pytest。
- 使用 `compileall` 检查全部 Python 模块。
- 自动发现并对全部 `src/governance_app/static/*.js` 执行 `node --check`。
- 检查 `scripts/start.sh` 和 `scripts/build_app.sh` 的 Shell 语法。

新增 GitHub Actions 质量工作流，在 Pull Request 和 `main` 推送时使用 Python 3.12 与 Node.js 运行 `scripts/check.sh`。现有 Windows 构建工作流继续独立运行，避免质量检查和平台打包相互掩盖失败原因。

本轮不引入 Ruff、类型检查和覆盖率阈值；这些工具应在后续工程治理阶段单独建立基线，避免一次变更混入大量无关格式修改。

## 测试策略

所有生产行为遵循测试先行。

### 迁移测试

- 全新数据库初始化到当前版本。
- 模拟版本 1 数据库升级且保留现有数据。
- 重复初始化不重复应用迁移。
- 迁移失败回滚且不记录版本。
- 高于应用版本的数据库拒绝启动。

### 备份恢复测试

- 在线备份包含活动数据库数据。
- 生成的备份通过完整性检查。
- 损坏文件拒绝恢复且活动数据不变。
- 恢复成功后数据与来源一致。
- 恢复过程失败时自动还原安全备份。

### 稽核测试

- 新问题首次生成。
- 持续命中时关联最新 `audit_result`。
- 不再命中时标记 `resolved_by_reaudit`。
- 再次命中时重新打开。
- 人工整改值和说明在持续命中与重新打开时保留。
- 专题分析不读取已由重复稽核解除的问题。

### 服务测试

- 重操作冲突返回 409。
- 操作完成或异常后锁均释放。
- 正常大小上传继续可用。
- 超过 100 MiB 的请求在读取正文前返回 413。
- 非法或缺失请求长度返回 400。

### 回归验证

- 现有全部测试继续通过。
- 所有 JavaScript 文件通过语法检查。
- Python 源码通过全量编译。
- 启动和构建 Shell 脚本通过语法检查。

## 兼容性与发布

- 保留当前工作区目录结构、数据库文件名和备份目录。
- 旧数据库首次由新版本打开时自动升级；升级前由迁移流程生成安全备份。
- API 成功响应、Excel 输入输出和前端操作流程不变。
- 应用版本在实现完成后从 `0.1.0` 提升到 `0.2.0`，并在版本接口中暴露新的数据库版本。
- Windows 打包仍生成单文件可执行程序。

## 验收标准

- 版本 1 数据库可以无数据丢失地升级到当前版本。
- 损坏备份无法覆盖活动数据库。
- 重复稽核后问题内容与最新结果一致，已解除问题不会进入当前专题分析。
- 并发重操作得到确定的 409 响应，进程不会进入永久锁定状态。
- 超大上传在分配完整请求正文前被拒绝。
- 本地检查和 PR 质量工作流覆盖全部测试、Python 模块、JavaScript 模块和 Shell 脚本。
- 现有业务流程和 145 个基线测试保持兼容。
